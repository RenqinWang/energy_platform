# 作业2要求对照与待完善清单

更新时间：2026-05-24  
对照文件：`docs/作业2-分布式数据分析.pdf`

## 结论

当前平台的主体链路已经基本成型：三节点 HDFS、Kafka 接入、Delta Lake 三层数据湖、Spark 批处理/模拟流式微批、FastAPI、React 前端，以及冷机、热机、冷热电三联供的 Gold 报表、预测和收益展示。

但按作业 PDF 的验收口径看，仍有几类内容需要补齐或收口：

1. 文档口径不统一，旧文档仍保留“3 台冷机、3 万行数据、XGBoost、100% 完成”等过期说法。
2. 分布式有效性验证目前偏弱，现有报告主要是 `local[1]` 与 `local[*]` 对比，最好补一次真实 Spark standalone 集群验证或明确说明限制。
3. 价格数据与 2018 年能源数据时间不重合，收益预测需要明确“价格对齐/业务映射”口径。
4. 热机、冷热电三联供从 Silver 到 Gold 已有统一 Gold 结果，但 Silver 层设备宽表和字段完整性说明还不够清楚。
5. 供能趋势预测已经有随机森林方案，但“特征设计、时间窗口、训练评估、结果解释”需要整理成可答辩的正式文档，并说明是否只有一个算法或是否补充基线算法。
6. 前端“实时更新”“在线/离线”等概念需要统一成模拟流式/运行状态口径，避免被问成真实设备心跳。

## P0：提交前必须收口

### 1. 统一最终文档口径

当前文档存在明显冲突：

- `docs/archive/PROJECT_COMPLETION_SUMMARY.md` 仍写“32,292 条、3 个冷机设备、75 天运行数据”，已经不是当前口径。
- `DATA_COMPLETENESS_ANALYSIS.md` 写的是 6,117,217 条、冷机状态宽表 403,820 条，是较新的口径。
- `docs/archive/FINAL_COMPLETION_REPORT.md` 写“总体完成度 100%”，但它没有覆盖后续 full/stream 切换、真实微批增量、热机/三联供统一 Gold 的最新工作。
- `docs/archive/FORECAST_AND_ADVICE_IMPLEMENTATION.md` 写 XGBoost，而当前前端文档和实现口径已经偏向 RandomForestRegressor。

建议新增或重写一个最终说明文档，例如：

```text
docs/FINAL_SYSTEM_DESCRIPTION.md
```

建议结构：

1. 项目目标与业务场景
2. 三节点部署架构
3. 数据源说明：Kafka 采集数据、价格 API
4. 数据湖设计：full 与 stream 两套命名空间
5. Bronze/Silver/Gold 表设计和字段说明
6. 批处理与模拟流式微批流程
7. 综合报表计算方法
8. 供能趋势预测方法
9. 收益预测与价格对齐策略
10. 前端页面与接口说明
11. 分布式有效性验证
12. 组员分工、贡献比例、运行方式、演示步骤

### 2. 补清楚价格数据与能源数据的时间对齐

作业数据有一个天然矛盾：

- 能源采集数据是 2018 年 1、2、4、7、10 月。
- 价格 API 是 2026 年每日更新的价格。

收益预测如果直接把 2026 价格套到 2018 供能预测上，需要明确为“业务价格映射”：

```text
能源负荷使用 2018 年历史工况；
价格使用当前价格 API 的最新价格或同月同日映射价格；
收益结果表示在当前价格政策下复盘/预测该类负荷的经济性。
```

需要在文档和前端说明里避免写成“2018 年真实收益”。

### 3. 补强任务五分布式验证

现有 `distributed_effectiveness_validation.md` 的价值是有的，但报告自己也承认 Spark standalone 远端 Worker 未检测到，所以它更像“本机并行 Spark + HDFS”验证。

建议优先做其中一种：

- 性能对比：同一脚本分别跑 `local[1]`、`local[*]`、`spark://node2:7077`，记录耗时、输入行数、输出 checksum。
- 容错验证：停止一个 DataNode 或 Kafka-to-Bronze，再恢复，展示 HDFS 无丢块、微批可继续。
- 扩展性验证：提高 Kafka 生产速率或模拟更多站点/设备，观察 Bronze、Silver、Gold 微批耗时变化。

最稳妥的是性能对比，因为已有脚本和结果，只需要补真实集群模式。

## P1：功能与数据口径建议完善

### 1. 热机、三联供的 Silver 说明需要补强

当前冷机有比较明确的 `silver_chiller_status`，字段包括供回水温、压力、流量、运行状态、估算功率等。

但热机和三联供更多是通过统一 `silver_point_fact` 到 `gold_system_supply_hourly` 的路径完成，文档上容易被问：

- 热机的原始流量、温差、累计热量字段来自哪些点位？
- 三联供的冷、热、电分别如何从累计量变成小时供能量？
- 累计量重置、跳变、负差分怎么处理？
- 缺失值为什么为空，是否合理？

建议补一个表：

| 系统 | 原始点位类型 | Silver 字段 | Gold 指标 | 处理规则 |
|---|---|---|---|---|
| 冷机 | 温度、流量、压力、运行状态 | supply_temp/return_temp/flow/pressure/run_flag | cooling_supply_kwh | 水侧公式和 COP 估算 |
| 热机 | 累计热量、温度等 | point_fact 明细或 heating 状态口径 | heating_supply_kwh | 累计量按小时差分 |
| 三联供 | 累计冷量、累计热量、发电量等 | point_fact 明细或 cchp 状态口径 | cooling/heating/electric_supply_kwh | 按小时差分并汇总 |

### 2. 供能预测文档要从“能跑”变成“能答辩”

PDF 明确要求说明：

- 特征设计
- 时间窗口构造
- 模型选择依据
- 训练与评估方式
- 结果分析与解释

建议最终文档明确写：

```text
目标变量：下一小时或未来 24 小时 supply_kwh
时间特征：月份、星期、是否周末、小时、季节
历史负荷：lag_1h、lag_24h、rolling_mean_24h、rolling_max_24h
设备状态：operation_rate、recent_24_supply、recent_24_operation_rate
训练方式：按时间顺序 80/20 划分，避免未来数据泄漏
模型：RandomForestRegressor，适合非线性负荷关系、小样本鲁棒、解释成本低
评估：MAE、RMSE、MAPE 或 R2
解释：当前停机设备的预测曲线表示趋势需求，不强制归零
```

如果时间允许，补一个简单基线算法更好：

```text
seasonal_hourly_baseline = 同系统/同小时/同月份历史均值
```

这样前端“不同算法下的趋势预测结果”更容易解释。

### 3. 前端状态语义需要改口径

作业要求“设备状态信息”和“实时更新能力”，但我们的数据不是设备真实心跳，而是历史数据通过 Kafka 模拟重放。

建议统一：

- “在线”改为“运行”
- “离线”改为“停机”
- “告警”保留为数据质量或异常指标
- 页面提示写成“模拟流式更新，数据来自 Kafka 重放并经 5 分钟微批刷新”

这样可以避免答辩时被问“2018 年数据为什么能判断 2026 年在线”。

## 当前可以认为已经满足的部分

- 三节点部署架构已经具备，HDFS 有三 DataNode 口径。
- Kafka 传感器数据接入和价格 API 接入已经实现。
- Delta Lake Bronze/Silver/Gold 三层已经实现，并进一步拆分 full/stream 命名空间。
- Gold 层已面向前端提供综合报表、预测、收益预测等数据产物。

- 微批增量方向已经明确：Kafka latest offset、Silver ingest watermark、last-batch 明细表、无新增跳过下游 Gold。

## 最终判断

平台实现层面已经接近可提交状态，但文档层面不能直接使用现有多个“完成报告”拼接提交。下一步最应该做的是整理一份最终统一说明文档，并把分布式验证、价格对齐、预测方法、热机/三联供字段映射这四块写扎实。
